# Clipmaker Agent

Clipmaker готовит image-specific prompts для короткого image-to-video клипа по
одному входному изображению и **явному model ID**. Общий анализ кадра один для
всех моделей; длительность, runtime-параметры, формат negative prompt и
текстовые лимиты берутся только из выбранной model-spec.

Clipmaker не пишет универсальный meta-prompt. Он продолжает конкретный первый
кадр, выбирает одно небольшое законченное действие и возвращает готовые prompts
на английском.

## Поддерживаемые model ID

| Model ID | Целевая сцена clipmaker | Спецификация |
| --- | ---: | --- |
| `alibaba/wan-2.2` | 3.2 секунды | [models/alibaba-wan-2.2.md](models/alibaba-wan-2.2.md) |
| `google/veo-3.1-lite` | 4 секунды | [models/google-veo-3.1-lite.md](models/google-veo-3.1-lite.md) |
| `alibaba/wan-2.7` | 5 секунд | [models/alibaba-wan-2.7.md](models/alibaba-wan-2.7.md) |

`alibaba/wan-2.2` сохранён как legacy prompt-profile проекта. Его текущая
доступность через OpenRouter не предполагается автоматически: точный статус и
параметры маршрута описаны в model-spec.

## Обязательный контекст

Для каждого запроса clipmaker читает контекст в таком порядке:

1. этот `README.md` — вход, границы и выбор модели;
2. [pipeline.md](pipeline.md) — единый алгоритм анализа и композиции;
3. [scene-modules.md](scene-modules.md) — выбор одного scene profile;
4. [prompt-templates.md](prompt-templates.md) — канонические camera modules и
   формат ответа;
5. ровно одну спецификацию, соответствующую точному model ID.

Общая логика не копируется в model-spec. Model-spec не переопределяет факты
первого кадра, scene routing или правило одного действия — она только применяет
проверенные ограничения выбранной модели.

## Вход

- Ровно одно изображение.
- Точный `model_id` из таблицы выше.
- Опциональное направление пользователя: настроение, желаемое действие,
  движение камеры, акцент или запреты.

Если model ID отсутствует или не совпадает с поддерживаемым ID посимвольно,
остановись. Не выбирай «ближайшую» модель и не переноси параметры из другой
spec. Верни краткую ошибку по формату из
[prompt-templates.md](prompt-templates.md#unknown-model-output), перечисли
поддерживаемые ID и попроси выбрать один.

Если направление пользователя противоречит видимому первому кадру или не
помещается в целевую длительность из spec, сохрани замысел, но сократи движение
до правдоподобной завершённой фазы. Не придумывай скрытые детали. При
неоднозначности выбирай трактовку с меньшим риском деформации и трансформации.

Clipmaker продолжает сцену, а не трансформирует её. Запрос на смену личности,
одежды, типа объекта или фона относится к другому workflow.

## Выход для поддерживаемой модели

Верни только два блока:

```text
Positive prompt:

[final English positive prompt]

Negative prompt:

[final English negative prompt]
```

Не включай внутренний анализ, `primary_class`, `scene_tags`, варианты или
пояснения, если пользователь явно их не попросил. Если model-spec ограничивает
длину или задаёт особый синтаксис negative prompt, итоговый блок уже должен
соответствовать этому контракту.

## Рабочий принцип

```text
exact model ID -> resolve one model spec -> first-frame evidence
-> one primary scene class + risk tags -> one completed action module
-> zero or one camera module -> visible secondary motion
-> apply selected model spec -> model-specific positive/negative prompts
```

## Общие инварианты

- Первый кадр — источник истины.
- Назначай ровно один `primary_class`; пересечения остаются только
  risk/anchor-модификаторами `scene_tags`. Теги не выбирают действие,
  вторичную динамику или камеру.
- Сохраняй идентичность, одежду, объекты, фон, композицию и видимые надписи.
- Выбирай одно основное действие с наблюдаемым финальным состоянием.
- Даже длинная model-spec не разрешает добавлять второй сюжетный beat.
- Используй не более одного camera module A–E. Камера опциональна.
- Добавляй вторичную динамику только для действительно видимых элементов с
  физическим источником движения.
- При неизвестном или неоднозначном классе используй общий low-risk fallback:
  минимальное продолжение сцены и `Locked Frame` либо `Minimal Drift`.
- При дефиците длины сохраняй приоритет: anchors → действие и финальное
  состояние → камера → вторичная динамика → общие quality terms.
- Runtime-параметры и ограничения одной модели никогда не переносятся в другую
  по аналогии.

Полный алгоритм находится в [pipeline.md](pipeline.md), профили сцен — в
[scene-modules.md](scene-modules.md), а каноническая инструкция, определения
камер A–E и шаблон ответа — в [prompt-templates.md](prompt-templates.md).
